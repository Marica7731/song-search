# AGENTS.md

本仓库任务请遵守以下要求。

## 1. 工作目录

- 开始任何文件读写、构建、测试、发布、commit 前，先确认当前目录是正确的项目根目录。
- 始终以当前项目根目录作为工作目录。
- 不要使用乱码路径、错误路径或误编码用户目录路径，例如 `C:\Users\����`。
- 如果路径不确定，请先停止操作，并说明需要确认项目根目录。

## 2. Git Commit

- 只要本轮进行了代码、文档、配置或资源文件的落地修改，完成后请整理一次规范 commit，除非用户明确要求不要提交。
- commit 前检查 `git status` 和 `git diff`，确认只包含本次任务相关修改。
- 不要提交临时文件、缓存文件、构建产物、日志文件、无关配置文件。
- 不要提交 token、cookie、AI key、二维码、真实运行态 JSON 或其它敏感信息。
- commit message 使用清晰中文，说明本次修改目的，例如 `feat: 增加xxx`、`fix: 修复xxx`、`docs: 更新xxx说明`。

## 3. 文档

- 涉及新功能、行为变更、部署/发布流程、排障流程或后续维护成本较高的改动时，请更新 `README.md` 或 `docs/` 下的说明文档。
- 文档需要包含功能说明、使用方法、文件说明、注意事项和测试说明。
- 纯只读排查、简单错字修复、临时验证、用户明确要求不写文档时，可以不新增文档，但最终回复要说明原因。

## 4. 文件说明

- 如本轮新增或修改多个文件，请用“文件清单 / file manifest”或“项目结构说明 / project structure”形式描述主要文件。
- 每个主要文件至少说明文件路径、文件用途、主要函数或模块职责、与其他文件的关系。
- 单文件小修可以简化说明，但仍要说清改了什么、为什么改。

## 5. 输出

最终回复请总结：

- 修改了哪些文件。
- 每个文件的作用。
- 如何运行。
- 如何测试。
- 是否已发布、重启、发版或推送到用户可见环境。
- commit message 是什么。

如果没有代码修改或没有 commit，请明确说明原因。如果无法完成线上发布或真实验证，请明确说明阻塞原因和下一步。

## 6. 目标、并行智能体与收尾

- 对非简单的一步任务，尤其是涉及多个模块、排障取证、Web/App/客户端/服务端/后台任务、发布验证或真实链路的任务，请先为本轮工作设定一个清晰的新目标，并拆成可验收的子任务。
- 如果当前工具支持 goal / thread goal，请在开始执行前创建或更新目标；目标必须写清真实完成标准，不要只写“分析一下”。
- 如果当前工具支持子智能体/多智能体，本条即为用户对所有非简单任务的显式授权和显式要求：请默认使用子智能体并行处理彼此独立的部分，以加快速度为核心，除非用户在当前任务中明确说不要开子智能体。
- 每个子智能体都要有专属目标；如果环境支持 `/goal`，为每个子智能体设置独立 `/goal`，否则在分配 prompt 中明确写出目标、范围、禁止事项和交付格式。
- 主线程不要空等子智能体；等待期间继续推进不重叠的工作。子智能体返回后必须整合、复核和取舍，不要无脑照收。
- 子智能体完成、取消或不再需要后，必须及时 close / 清理，避免长期占用并发槽、污染侧边栏或留下无用运行态。
- 历史遗留但已经完成/关闭的子智能体一般不用管；只有仍处于运行、等待或占用资源状态的，才需要主动关闭。
- 简单问答、单条命令、小范围文案或纯确认任务不要强行开 goal/子智能体，避免形式化开销。

## 7. 真实交付与可见验证

- 用户可见的功能改动，不能只停留在本地运行成功、单元测试通过或“代码已改”。应根据项目实际交付方式完成发布、重启、部署、发版、安装、推送或配置更新。
- 涉及前端页面、客户端、服务端、后台任务、自动化流程、通知、上传/下载/投稿等真实链路时，需要用可观察证据验证效果，例如页面截图、API 返回、日志、队列状态、运行进程、线上服务状态、设备安装结果或最终业务数据。
- 如果项目有既定发布脚本、服务管理方式、发版流程或设备推送流程，应优先使用项目已有流程，不要临时发明一套发布方式。
- 如果运行中任务、权限、网络、外部服务或用户确认阻止了发布、重启或真实验证，最终回复必须明确说明已完成什么、还没上线什么、阻塞原因、建议的下一步。
- 不要把“本地测试通过”表述成“用户已经能看到效果”，除非已经完成对应的用户可见环境验证。
